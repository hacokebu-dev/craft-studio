‘출판 편집자를 위한 올인원 업무 툴킷’이라는 컨셉으로 웹 서비스 Tools를 만들었어요. 이 프로젝트는 출판 편집자 커뮤니티 ‘에디터스 룸’의 제안에서 시작됐죠. 커뮤니티를 개설하고 나니 이런 고민이 들었다고 해요. “사람들이 계속 들어오게 만들 계기가 필요하다.” 그 답으로 떠올린 게, 출판 편집자의 업무에 도움이 될 웹 도구들을 모아보자는 아이디어였어요. 그리고 그 아이디어의 구현을 하코 앤 케부에게 제안한 것이었죠.
&nbsp;

&nbsp; 

## 기획은 현업의 목소리에서 시작했어요
<img src ="/assets/tools_19.jpg">
출판 편집자의 업무를 잘 모르는 하코 앤 케부가 상상으로 도구를 만들 수는 없었어요. 그래서 실제 출판 편집자들과 인터뷰를 진행했고, 그 과정을 통해 편집 도구 6종을 기획했어요. 그리고 편집자들과 이야기하며 떠올린 게 하나 있었어요. 편집자도 결국 보통의 직장인이라는 점이었죠. 그래서 편집 업무 도구 외에도 ‘근무 기간 계산기’와 ‘연봉 실수령액 표’ 같은, 일하면서 은근히 자주 찾게 되는 도구도 개발하기로 했어요. 여기에 더해, 편집자들에게 도움이 될 만한 사이트들을 모은 북마크 페이지도 제공하기로 했고요.
&nbsp;

&nbsp; 

## 기획 → 프로토타입 → 리뷰 → 개선

기능에 대한 상세 기획은 직접 했고, 프로토타입을 만든 뒤 편집자들에게 리뷰를 받았어요. 리뷰를 통해 꽤 중요한 개선점들이 나왔어요. 예를 들자면,
<img src ="/assets/tools_20.jpg">
- 책등 계산 결과와 글자 수 계산 작성 내용을 바로 복사할 수 있는 기능.
- 글자 수 계산 시 원고지 장수 표시.
- QR 코드 생성 시 이미지 확장자 안내.

같은 것들이에요. 상상으로 만들었다면 놓쳤을 법한 부분들이었죠.
이렇게 리뷰 → 개선 → 반영의 과정을 여러 번 거쳐 지금의 형태가 됐어요.
&nbsp;

&nbsp; 

## 기술은 최대한 단순하게

이번 프로젝트에서는 일부러 기술 스택에 대한 욕심을 버렸어요. 서비스의 스펙을 놓고 봤을 때, 굳이 최신 기술을 써야 할 이유가 없어 보였거든요. 그래서 HTML, CSS, JavaScript만 사용했고 JavaScript도 가능하면 바닐라 JS로만 작성했어요. 유지 보수 이슈를 최소화하고 싶었거든요.
<img src ="/assets/tools_21.jpg">
저장이 필요한 부분은 로컬 스토리지를 사용했고, 북마크처럼 데이터 관리가 필요한 부분은 JSON 파일로 관리했어요. 이 역시 유지 보수를 쉽게 하기 위한 선택이었죠.
&nbsp;

&nbsp; 

## 디자인은 간결하고, 재사용 가능하게
<img src ="/assets/tools_22.jpg">
디자인에서도 기준은 명확했어요. 간결함과 재사용성.

한 번 개발한 디자인 요소는 다른 페이지에서도 그대로 쓸 수 있길 원했어요. 그래서 어떤 도구가 추가되어도 디자인이 복잡해지지 않고 일관성을 유지할 수 있도록 설계했어요.
&nbsp;

&nbsp; 

## 배포 방식은 결국 GitHub로

처음에는 기술에 익숙하지 않은 사람도 편하게 관리할 수 있도록 배포를 Netlify Drop 방식으로 결정했어요. 그런데 써보니 문제가 있었어요. 특정 파일만 서버에서 수정하거나 교체할 수 없고, 조금이라도 변경이 있으면 전체 서비스를 다시 업로드해야 했죠. 이게 개발 과정에서는 매우 불편해서 결국 GitHub에 연결해서 배포하는 방식으로 바꾸게 됐어요.
&nbsp;

&nbsp; 

## 재밌었던 부분: 책 크기 비교 테이블

가장 재밌게 고민했던 건 책 크기 비교 기능이었어요. 숫자로만 보여주면 기존 자료랑 다를 게 없고, 이미지만 보여주기엔 직관성이 떨어져서 고민이었죠. 그러다 떠올린 방식이 테이블에 마우스를 올리면 해당 판형이 시각적으로 보이는 구조였어요. 
<img src ="/assets/tools_23.jpg">
이걸 기술적으로 어떻게 풀어야 할지도 고민했는데, 결국 가장 단순한 방식인 판형별 이미지를 하나씩 만들어 두고, 호버 시 해당 이미지를 보여주는 형태로 구현했어요. 그럼에도 사용자 입장에서는 꽤 재밌는 인터랙션으로 보일 거 같아 만족스러웠어요. 
&nbsp;

&nbsp; 

## 가장 어려웠던 부분: 데이터 수집

돌이켜보면, 기술보다 더 어려웠던 건 데이터 수집이었어요.

책등 계산을 위해선 종이 두께의 정보가 필수인데, 종이는 대부분 무게만 공식적으로 제공하고, 두께 정보를 알려주진 않더군요. 그래서 두께 정보가 웹에 공개된 유명한 종이만 추가할 수 있었어요.

판형별 대표 책 선정도 어려웠어요. 서점에서 판형 기준으로 책 정보를 제공하지 않으니, 책의 크기를 하나씩 대조하며 대표 책을 선정해야 했어요. 이 도구를 만든 이유 자체가 그대로 난이도가 된 거였죠.(다른 판형은 다 6종씩인데, 국배판,타블로이드 판형만 3종인 이유입니다. 못 찾았어요...)

마지막으로 툴스는 위에서 말했듯 유지 보수를 위해 데이터를 JSON으로 관리하는데요, 하나씩 추가하는 것 자체는 어렵지 않지만 초기 데이터를 만들어 넣는 과정은 정말 번거로웠어요.
&nbsp;

## 마치며

Tools는 화려한 기술로 만든 서비스는 아니지만 실제로 잘 쓰일 수 있는가를 가장 중요하게 생각하며 만들었어요. 그래서 편집자들의 웹브라우저에 툴스가 한 자리를 차지했으면 좋겠어요.
